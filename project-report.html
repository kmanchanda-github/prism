Context
The Problem
Product engineers lose hours per incident manually collecting and correlating data before any analysis begins.
⏱️
Manual Correlation
Engineers manually pull logs from Splunk/Datadog, cross-reference Jira defects, and read code before writing any analysis.
🔍
Inconsistent Quality
Analysis quality varies by engineer experience. Junior engineers miss signals that experienced ones would catch.
📤
Sharing Overhead
Formatting analysis for different audiences (engineering, leadership, customers) requires re-writing the same content multiple times.
Goal: Given an incident description and metadata, automatically query the right sources in parallel, synthesize findings into a structured report, and deliver it to the engineer with options to chat, edit, approve, and share — all from one UI.
Design

Flows
Incident Intake
Engineers can submit manually via the UI, or incidents can be auto-ingested from Jira and other systems via webhooks.

🖥️ Path A — Manual (UI Form)

1
Engineer fills title, description, severity, metadata
2
POST /api/analysis → immediate job_id response
3
Browser polls every 3s while analysis runs
4
Report appears in UI on completion

⚡ Path B — Automatic (Webhook)

1
Jira creates/transitions a ticket matching trigger rules
2
Jira fires webhook → HMAC validated
3
Adapter maps payload to Incident model → Celery task
4
Slack/email sent with link: /analysis/{id}
Trigger rules are config-driven in config.yaml — only specific issue types, priorities, and project keys fire analysis, preventing noise from low-priority tickets.

Features
Key Capabilities
Beyond automated analysis — the system supports a full engineer review workflow with AI collaboration.
🤖
Multi-Agent Analysis Pipeline
Orchestrator LLM decides which agents to activate based on incident type. Log, code, and defect agents run in parallel and return structured findings. Synthesizer produces root cause, workaround, and prioritized actions.
💬
Chat with AI for Clarifications
Engineers can ask follow-up questions on any finding. The AI has full context — incident, raw sub-agent data, and current report. Responses stream in real time via SSE. AI can suggest direct edits to report fields (accept/dismiss).
📝
Analysis Versioning
Every edit (manual or chat-suggested) creates a new version. Original AI output (v0) is never overwritten. Version history with diffs is visible in the UI. Share and export always use the latest approved version.
📊
Multi-Audience Export
Four export templates: PDF Technical (full analysis + log references), PDF Executive (plain-language summary), PPT Executive (5-slide leadership deck), PPT Customer (incident report for external sharing).
🔌
Pluggable Adapters
Every external system is behind an abstract base class. Add Splunk, Datadog, Salesforce, or any other source by subclassing one file and adding a config.yaml entry — no changes to the pipeline.
📈
Token Usage Tracking
Every analysis records per-agent token usage and estimated cost. Aggregated trends identify which agents are most expensive over time, enabling targeted prompt optimization.

API
REST Endpoints
All routes are served by FastAPI. The React SPA is served from the same process as static files.
POST /api/analysis Submit incident (manual path)
GET /api/analysis/{id} Status + current report version
PATCH /api/analysis/{id} Save edited version
GET /api/analysis/{id}/versions All versions with diff data
POST /api/analysis/{id}/chat Send message → SSE stream response
GET /api/analysis/{id}/chat Full conversation history
POST /api/analysis/{id}/export Generate PDF or PPTX download
POST /api/analysis/{id}/action Execute workaround / share / notify
POST /webhooks/jira Jira webhook intake (HMAC validated)
POST /webhooks/salesforce Salesforce case event intake
POST /webhooks/generic Standard payload for any other system

Getting Started
Running the Project
One docker-compose command starts the full backend. The React dev server proxies API calls automatically.
🐳 Docker (recommended)
cp .env.example .env
# Set ANTHROPIC_API_KEY in .env

docker-compose up -d
# Starts: Postgres · Redis · API · Celery

cd ui && npm install
npm run dev
# UI on :5173 · API on :8000
🛠️ Local Dev
uv pip install -e ".[dev]"

# Run API
uvicorn src.api.main:app --reload

# Run worker (separate terminal)
celery -A src.api.worker.celery_app \
  worker --loglevel=info

# Migrations
alembic upgrade head
