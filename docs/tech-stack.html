FastAPI v0.115

Modern, high-performance Python web framework for building APIs with automatic OpenAPI docs, built-in async support, and Pydantic-powered validation.

FastAPI sits at the top of the Python async web stack. You declare endpoints with plain Python functions and type hints; FastAPI handles request parsing, validation, serialisation, and generates interactive docs — all automatically. Under the hood it runs on Starlette (ASGI) and Pydantic. Benchmarks consistently put it among the fastest Python frameworks, on par with Node.js and Go for I/O-bound workloads.
Key Concepts

Core building blocks

Path operations
@app.get(), @app.post() etc. declare HTTP routes. The decorated function's return value is serialised to JSON automatically.
Pydantic models
Request bodies and responses are defined as Pydantic BaseModel subclasses. FastAPI validates inputs and rejects bad requests with a 422 before your code runs.
Dependency Injection (Depends)
Reusable logic (auth, DB sessions, config) is expressed as dependencies injected into endpoint function signatures via Depends().
async def vs def
Use async def for I/O-bound endpoints (DB queries, HTTP calls). FastAPI runs sync endpoints in a thread pool automatically.

Useful features

Auto docs
/docs (Swagger UI) and /redoc are generated automatically from your route definitions and Pydantic models. Zero extra work.
StreamingResponse / SSE
Return a StreamingResponse with an async generator to push data chunk-by-chunk. Prism uses this for chat token streaming.
StaticFiles
Mount a directory as static files. Prism mounts ui/dist/ so FastAPI serves the compiled React app — single container, no separate web server.
Background Tasks
Lightweight post-response work without a full queue. For heavy jobs, delegate to Celery instead.
How Prism Uses It

Prism's API layer

📬All REST routes live under src/api/routes/ — analysis CRUD, chat SSE, webhook intake, export, and action approval each in their own file.
🌊The /api/analysis/{id}/chat endpoint returns a StreamingResponse via sse-starlette, pushing each LLM token as a data: {...} line the React client consumes in real time.
📁app.mount("/", StaticFiles(directory="ui/dist", html=True)) serves the compiled React SPA. No Nginx needed in development or simple deployments.
💉get_db() is a FastAPI dependency that yields an AsyncSession and closes it after the response — consistent across every DB-touching route.
Code Snapshot
src/api/routes/analysis.py — POST /api/analysisPython
@router.post("/api/analysis", response_model=AnalysisResponse)
async def create_analysis(
    payload: AnalysisRequest,             # Pydantic validates the body
    db: AsyncSession = Depends(get_db),  # injected DB session
):
    report = AnalysisReportORM(**payload.model_dump())
    db.add(report)
    await db.commit()

    task = run_analysis_task.delay(str(report.id))
    return AnalysisResponse(job_id=task.id, report_id=report.id)

LangGraph v0.2

Framework for building stateful, multi-actor LLM workflows as directed graphs. Each node is a Python function; edges define control flow; a typed State object flows through the graph.

LangGraph models AI pipelines as directed graphs rather than linear chains. This unlocks parallelism (fan-out to multiple agents simultaneously), conditional routing (send to log agent only if logs are available), human-in-the-loop pauses (interrupt before executing a risky action), and full state persistence across long-running workflows. It is the backbone of Prism's analysis pipeline.
Key Concepts

Graph primitives

StateGraph
The computation graph. You define it with a typed State class, add nodes, add edges, then call .compile() to get a runnable.
Node
A Python function (sync or async) that receives the current State, does work, and returns a dict of state updates. Nodes are the units of work.
Edge & Conditional Edge
add_edge(A, B) always goes A→B. add_conditional_edges(A, router_fn) calls router_fn with state and routes to the returned node name.
Send API (fan-out)
Return a list of Send(node, state) objects from a node to invoke that node in parallel with different inputs — how Prism runs log, code, and defect agents concurrently.

Advanced features

State & Reducers
State is a TypedDict or Pydantic model. Annotate list fields with Annotated[list, operator.add] so parallel nodes append rather than overwrite.
Checkpointer
Persist the full graph state to Postgres (AsyncPostgresSaver) or memory after every step. Enables resuming long workflows after a crash.
interrupt()
Pause execution at a node and surface the current state to a human. Execution resumes when the human approves via Command(resume=...). Used for Prism's workaround approval gate.
Subgraph
Compile a graph and use it as a node inside a parent graph. Enables composable, reusable pipelines.
How Prism Uses It

The analysis pipeline graph

🔀route_decision checks which data sources are available and returns a list of Send objects — one per applicable sub-agent — triggering them in parallel.
🔬Each of log_agent, code_agent, and defect_agent runs concurrently, appending its findings to state.agent_findings via a list reducer.
✅quality_check uses a conditional edge — if the synthesised report score is below threshold it loops back to synthesizer; otherwise it proceeds to notify.
💾An AsyncPostgresSaver checkpointer persists graph state after every node, so a Celery worker crash doesn't lose a partially complete analysis.
Code Snapshot
src/agents/orchestrator.py — graph definitionPython
from langgraph.graph import StateGraph, START, END
from langgraph.types import Send

def fan_out(state: AnalysisState) -> list[Send]:
    sends = []
    if state.log_bundle_path:
        sends.append(Send("log_agent",   state))
    if state.repo_url:
        sends.append(Send("code_agent",  state))
    if state.jira_project_key:
        sends.append(Send("defect_agent", state))
    return sends

builder = StateGraph(AnalysisState)
builder.add_node("parse_incident", parse_incident)
builder.add_node("log_agent",     log_agent_node)
builder.add_node("synthesizer",   synthesizer_node)

builder.add_edge(START, "parse_incident")
builder.add_conditional_edges("parse_incident", fan_out)
builder.add_edge("log_agent", "synthesizer")

graph = builder.compile(checkpointer=checkpointer)

LangChain v0.3— the abstraction layer LangGraph nodes call

LangGraph is built on top of LangChain. Every node that calls an LLM uses LangChain's unified chat model interface — the same .invoke() / .astream() API works across all providers.

Core primitives

Chat Models
ChatAnthropic, ChatOpenAI, ChatGoogleGenerativeAI, ChatBedrock — all share the same interface. Call .invoke(messages) or await .ainvoke(messages).
Messages
HumanMessage, AIMessage, SystemMessage. Build a conversation as a list and pass it to the model. No provider-specific format needed.
PromptTemplate
Parameterise prompts with variables. .format_messages(incident=...) fills the template at runtime without string concatenation.
UsageMetadata
response.usage_metadata exposes input_tokens and output_tokens. Prism's TokenTracker reads this after each call to accumulate cost.

Invocation patterns

.invoke() — sync
Blocks until the full response arrives. Returns an AIMessage. Used by the synthesizer and analyst nodes.
.ainvoke() — async
Async equivalent. Use inside async def LangGraph nodes to avoid blocking the event loop.
.astream() — async streaming
Yields AIMessageChunk objects token by token. Powers Prism's chat SSE endpoint — each chunk is forwarded directly to the browser.
How Prism Uses It

The get_llm() factory

🏭src/core/llm.py reads LLM_PROVIDER and LLM_MODEL from env, lazy-imports only the relevant langchain-* package, and returns the appropriate chat model object.
🔄Every LangGraph node calls get_llm() — the returned object has the same .invoke() / .astream() interface regardless of provider. Switching providers requires only two env var changes.
💬The chat agent calls llm.astream(messages) and yields each chunk's .content into the SSE response for real-time streaming.
src/core/llm.py — LangChain provider factoryPython
def get_llm(*, max_tokens: int | None = None, temperature: float | None = None):
    provider = settings.llm_provider   # e.g. "anthropic"
    model    = settings.llm_model      # e.g. "claude-sonnet-4-6"

    if provider == "anthropic":
        from langchain_anthropic import ChatAnthropic
        return ChatAnthropic(model=model, max_tokens=max_tokens,
                              api_key=settings.anthropic_api_key)
    if provider == "openai":
        from langchain_openai import ChatOpenAI
        return ChatOpenAI(model=model, max_tokens=max_tokens,
                           api_key=settings.openai_api_key)
    # ... google, bedrock ...
    raise ValueError(f"Unknown LLM_PROVIDER='{provider}'")

Celery v5.4

Distributed task queue for Python. Offloads long-running jobs to background workers so the API responds instantly, with automatic retries, scheduling, and horizontal scaling.

AI incident analysis takes 2–15 minutes. If the API waited for it to finish, HTTP requests would time out and the user experience would be terrible. Celery solves this by decoupling job submission from job execution: the API enqueues a task (returns immediately with a job ID), and one of many background workers picks it up, runs it, and stores the result. Workers scale independently of the API — add more machines, handle more concurrent analyses.
Key Concepts

Core model

Task
A Python function decorated with @celery_app.task. Call task.delay(*args) to enqueue it. Celery serialises the args and pushes a message to the broker.
Broker
The message queue between producer (API) and consumers (workers). Prism uses Redis. RabbitMQ is the other common choice.
Worker
A process that connects to the broker, polls for tasks, executes them, and writes results to the result backend. Run with: celery -A app worker.
Result backend
Where task state (PENDING, STARTED, SUCCESS, FAILURE) and return values are stored. Prism uses Redis. AsyncResult(task_id).state polls the status.

Production settings

concurrency
Number of worker child processes (default = CPU count). Set to 100 for Prism's scale target. Each analysis runs in its own process.
worker_max_tasks_per_child
Restart the child process after N tasks. Prevents memory leaks from accumulating. Prism sets this to 50.
task_time_limit
Hard kill a task after this many seconds. Prevents runaway LLM calls from blocking workers forever.
Celery Beat
Scheduler for periodic tasks (cron-like). Runs as a separate process. Prism uses it for scheduled metric aggregation.
How Prism Uses It

The analysis task

📤When POST /api/analysis receives a submission, it calls run_analysis_task.delay(report_id) and immediately returns {job_id, report_id} to the caller.
⚙️The Celery worker picks up the task, runs the full LangGraph pipeline (which may take several minutes), and writes the resulting Markdown report as a new AnalysisVersionORM row.
🔄The React UI polls GET /api/analysis/{id} every 3 seconds (React Query). When status flips from running to complete, the report renders automatically.
📊Worker concurrency is set to 100 in docker-compose.yml, with worker_max_tasks_per_child=50 to keep memory footprint bounded.
Code Snapshot
src/api/worker.py — task definitionPython
from celery import Celery

celery_app = Celery("prism",
    broker=settings.redis_url,
    backend=settings.redis_url,
    broker_connection_retry_on_startup=True,
)
celery_app.conf.worker_max_tasks_per_child = 50

@celery_app.task(bind=True, max_retries=2)
def run_analysis_task(self, report_id: str):
    try:
        asyncio.run(_run(report_id))
    except Exception as exc:
        raise self.retry(exc=exc, countdown=30)

Redis v7

In-memory data store used as Celery's message broker and result backend. Sub-millisecond reads and writes, with optional persistence, pub/sub, and expiry.

Redis is an in-memory key-value store that can also act as a message queue (via lists), a pub/sub system, a cache, and a session store. Its speed comes from keeping all data in RAM. In Prism it has two roles: broker (Celery pushes task messages in; workers pop them out) and result backend (task state and return values stored as keys with TTL).
Key Concepts

Data structures

String
The simplest type — any binary-safe value. SET key value / GET key. Celery result backend stores task state here with an optional EX (expiry in seconds).
List
Ordered sequence with O(1) push/pop at both ends. Celery's broker uses LPUSH (producer) and BRPOP (blocking pop by worker) to implement the task queue.
Hash
Field-value map stored under one key. Efficient for storing structured objects like session data or task metadata.
Pub/Sub
PUBLISH channel message / SUBSCRIBE channel. Used for real-time event broadcasting. Not used in Prism today but available for Phase 3 notifications.

Operations & persistence

TTL (Time To Live)
EXPIRE key seconds sets an expiry. Keys are auto-deleted when they expire. Celery sets a TTL on result keys so the DB doesn't grow unbounded.
RDB Snapshots
Periodic point-in-time dumps to disk (redis.conf: save 900 1). Fast restart but may lose the last few seconds of data on crash.
AOF (Append Only File)
Every write command is appended to a log file. Survives crashes with minimal data loss. Slower writes, better durability.
Connection via URL
redis://localhost:6379/0 — host, port, database number (0–15). Prism reads REDIS_URL from env and passes it to both Celery broker and backend config.
How Prism Uses It

Broker + result backend

📬When run_analysis_task.delay(report_id) is called, Celery serialises the task message and does an LPUSH onto a Redis list named after the queue (default: celery).
👷Each worker process blocks on BRPOP for the queue list. When a message arrives, the first worker to pop it begins execution — natural load balancing with no extra config.
📊Task state (PENDING → STARTED → SUCCESS/FAILURE) is written to Redis string keys prefixed with celery-task-meta-{task_id}, with a TTL of 24 hours by default.

PostgreSQL v16

Open-source relational database. ACID-compliant, highly extensible, with native JSON support, full-text search, and a rich ecosystem — the industry default for structured data.

PostgreSQL stores every analysis report, every version, every chat message, and every token usage record in strongly-typed, indexed, relational tables. It handles transactions correctly — if saving a new analysis version fails partway through, the DB rolls back automatically and no partial data is committed. Its JSONB column type lets Prism store semi-structured agent findings alongside typed relational columns, without needing a separate document store.
Key Concepts

Relational fundamentals

Tables, rows, columns
Data is organised in tables with a fixed schema. Each column has a type (TEXT, INTEGER, TIMESTAMPTZ, UUID, JSONB) enforced by the DB engine.
ACID Transactions
Atomicity, Consistency, Isolation, Durability. All writes in a BEGIN/COMMIT block either fully succeed or fully roll back. Critical for multi-table updates.
Foreign Keys & JOINs
analysis_versions.report_id references analysis_reports.id. JOINs combine rows from multiple tables in a single query without N+1 round trips.
Indexes
B-tree index on frequently queried columns (e.g. report_id, created_at) dramatically speeds up SELECT queries. GIN indexes accelerate JSONB field queries.

PostgreSQL-specific features

UUID primary keys
gen_random_uuid() generates collision-resistant IDs distributed across shards — safer than sequential integers for public-facing APIs.
JSONB
Stores JSON as binary. Supports indexing (GIN), containment operators (@>), and path extraction (data->'key'). Prism stores agent_findings as JSONB.
TIMESTAMPTZ
Timestamp with time zone. Always store in UTC and let the application layer convert for display. Avoids DST bugs in multi-region deployments.
pg_trgm / Full-text search
Built-in trigram similarity and tsvector full-text search. Can search Markdown report content without Elasticsearch for Phase 2+.
How Prism Uses It

Prism's data model

📋analysis_reports — one row per incident submission: title, severity, description, status, source system, metadata.
📝analysis_versions — one row per report revision: version number, full Markdown content, agent findings (JSONB), editor (AI or user email), created_at.
💬chat_messages — ordered conversation history per analysis: role (user/assistant), content, timestamp.
📊token_usage — per-agent, per-analysis token counts and estimated cost in USD, used by the Phase 3 dashboard.

SQLAlchemy & Alembic v2 / v1.13

SQLAlchemy maps Python classes to database tables and generates SQL. Alembic tracks schema changes as versioned migration scripts — the standard ORM + migration stack for Python.

Writing raw SQL in application code is brittle and hard to maintain. SQLAlchemy provides an ORM (Object Relational Mapper) — you define models as Python classes and interact with the DB using Python objects and methods. Alembic sits alongside it to manage schema evolution: when you change a model, Alembic generates a migration script that upgrades (or rolls back) the database schema safely without manual SQL.
Key Concepts

SQLAlchemy ORM

DeclarativeBase
Base class all ORM models inherit from. Provides the metaclass magic that maps Python class attributes to DB columns.
Mapped[T] columns
SQLAlchemy 2.0 syntax: class id: Mapped[UUID] = mapped_column(primary_key=True). Type hints drive column type inference and IDE autocompletion.
AsyncSession
Async version of Session. Use async with AsyncSession() as session, then await session.execute(select(Model)). Required for FastAPI async endpoints.
select() / scalars()
Build queries with select(Model).where(Model.id == id). Execute with await session.execute(q), fetch results with .scalars().all() or .scalar_one().

Alembic migrations

alembic revision --autogenerate
Compares your ORM models to the current DB schema and generates a migration script with upgrade() and downgrade() functions. Always review before applying.
alembic upgrade head
Applies all pending migrations in order. Alembic tracks which have run in the alembic_version table. Safe to run multiple times (idempotent).
alembic downgrade -1
Rolls back the most recent migration. Useful in development; dangerous in production without a tested downgrade() function.
alembic/versions/
Each migration is a Python file with a unique revision ID, a down_revision pointer (forming a chain), and upgrade()/downgrade() functions.
How Prism Uses It

Async ORM + migrations

🏗️All ORM models live in src/models/: AnalysisReportORM, AnalysisVersionORM, ChatMessageORM, TokenUsageORM — each a DeclarativeBase subclass.
💉get_db() is a FastAPI dependency that yields an AsyncSession via AsyncSessionLocal and closes it after the response — all routes receive a fresh, auto-closed session.
⚡The async engine uses asyncpg (non-blocking Postgres driver). Alembic uses a separate sync connection (psycopg2) because Alembic's internals don't support async.
🔄Workflow for schema changes: update model → alembic revision --autogenerate -m "description" → review generated file → alembic upgrade head.

React & Vite React 18 / Vite 5

React is a declarative component-based UI library. Vite is its modern build tool — instant HMR in dev, optimised output for production. Together they power Prism's engineer-facing interface.

React lets you build UIs as a tree of components — reusable functions that return JSX (HTML-like syntax in JavaScript). When data changes, React re-renders only the affected parts of the DOM. Vite replaces Webpack with native ES module serving during development (near-instant HMR) and Rollup for production bundles. TypeScript adds static typing, catching errors before they reach the browser.
Key Concepts

React fundamentals

Components & Props
Functions that accept a props object and return JSX. Composable — build complex UIs from small, focused components. Props flow top-down (parent → child).
useState
Local component state. const [value, setValue] = useState(initial). Calling setValue triggers a re-render. Keep state as minimal as possible.
useEffect
Runs after render for side effects (data fetching, subscriptions, DOM manipulation). The dependency array controls when it re-runs.
Custom Hooks
Functions starting with "use" that encapsulate stateful logic. useAnalysis and useChat in Prism abstract away polling and SSE streaming.

Libraries in use

React Query (TanStack Query)
Server-state management. useQuery fetches data, caches it, and re-fetches on a configurable interval. Prism polls GET /api/analysis/{id} every 3s while status=running.
React Router v6
Client-side routing. BrowserRouter + Route components map URL paths to page components. "/" → SubmitPage, "/analysis/:id" → AnalysisPage.
react-markdown
Renders Markdown strings as React elements. ReportViewer uses it to display the AI-generated analysis report with proper formatting.
Vite proxy
vite.config.ts: proxy /api → http://localhost:8000. Requests to /api in dev hit the FastAPI backend; no CORS issues during development.
How Prism Uses It

The Prism frontend

📋SubmitPage — a form (title, severity, description, log bundle upload) that POSTs to /api/analysis, then navigates to the analysis page with the returned ID.
📊AnalysisPage — polls status via useAnalysis. Shows a spinner while running; renders ReportViewer + ChatPanel + ActionBar when complete.
💬useChat — opens an EventSource SSE connection to /api/analysis/{id}/chat, accumulates token chunks, detects suggested_edit payloads, and exposes them to ChatPanel.
📝ReportViewer — toggles between read-only markdown display and an editable textarea. On save, PATCHes the report and invalidates the React Query cache to refetch.

import { useQuery } from "@tanstack/react-query";

export function useAnalysis(id: string) {
  return useQuery({
    queryKey: ["analysis", id],
    queryFn:  () => fetch(`/api/analysis/${id}`).then(r => r.json()),
    // poll every 3 s while the job is still running

      ["pending", "running"].includes(data?.status) ? 3000 : false,
  });
}

Docker v26

Containerisation platform that packages your app and its dependencies into portable, reproducible containers. Docker Compose orchestrates multiple services locally with a single command.

Without Docker, setting up Prism locally means installing Postgres 16, Redis 7, the right Python version, uv, and Node — with exact version matching across machines. Docker packages each service into a container: an isolated, reproducible environment. docker compose up -d starts all five services (Postgres, Redis, API, Celery worker, Celery Beat) in one command, identically, on any machine.
Key Concepts

Core model

Dockerfile
A recipe file — each instruction (FROM, RUN, COPY, CMD) adds a layer to the image. Layers are cached, so only changed layers rebuild on the next build.
Image vs Container
An image is the snapshot (built from Dockerfile). A container is a running instance of an image. Multiple containers can run from the same image simultaneously.
docker-compose.yml
Defines multiple services, their images/build context, environment variables, ports, volumes, and inter-service dependencies. The source of truth for the local dev stack.
Volumes
Mount paths that persist data outside the container lifecycle. Prism mounts a named volume for Postgres data so the DB survives container restarts.

Networking & operations

Service networking
Containers in the same Compose project share a network and reach each other by service name. The API connects to Postgres at postgresql://postgres:5432/prism — not localhost.
Port mapping
ports: "8000:8000" maps host port 8000 to container port 8000. Only exposed ports are reachable from outside Docker. Internal communication uses the service name.
env_file & environment
env_file: [.env] injects all variables from .env into the container. Sensitive values stay out of docker-compose.yml and out of git.
depends_on + healthcheck
depends_on: postgres: condition: service_healthy ensures the API waits for Postgres to be ready before starting — avoids race-condition startup failures.
How Prism Uses It

Prism's Compose stack

🐘postgres — official postgres:16 image, named volume for data persistence, health-checked with pg_isready.
🔴redis — official redis:7-alpine image. Lightweight; alpine variant saves ~100 MB image size.
⚡api — built from the project Dockerfile. Runs uvicorn src.api.main:app --host 0.0.0.0 --port 8000. Port 8000 exposed to host.
🌿worker — same image, different CMD: celery -A src.api.worker.celery_app worker --concurrency=4. Scale to more workers by raising concurrency or running multiple replicas.
🕐worker-beat — same image, runs Celery Beat scheduler for periodic tasks. Only one instance should run (no duplicate scheduled jobs).

Dockerfile — multi-stage Python imageDockerfile
FROM python:3.12-slim AS base
WORKDIR /app

# Install uv for fast dependency resolution
RUN pip install uv

# Install dependencies (cached layer unless pyproject.toml changes)
COPY pyproject.toml .
RUN uv pip install --system -e ".[anthropic]"

# Copy source last — keeps source-change rebuilds fast
COPY src/ ./src/
COPY ui/dist/ ./ui/dist/

CMD ["uvicorn", "src.api.main:app", "--host", "0.0.0.0", "--port", "8000"]
