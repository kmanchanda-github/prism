Context
The Problem
Before any analysis is written, engineers spend hours collecting data manually from disconnected systems.
⏱️

Manual Correlation

Engineers pull logs from Splunk or Datadog, cross-reference Jira defects, and read code diffs before writing a single line of analysis.

📉

Inconsistent Quality

Analysis depth varies by engineer experience. Junior engineers miss signals experienced ones catch, creating uneven incident response quality.

📤

Sharing Overhead

The same findings get re-written multiple times — for engineers, leadership, and customers — at a cost of hours per incident.

Goal: Given an incident and metadata, automatically query the right data sources in parallel, synthesise findings into a structured report, and let engineers review, chat, edit, and share — all from one UI.
Context
Objectives
Concrete goals the system is designed to meet, derived directly from the problems above.
1
Ingest and correlate incident data from logs, code changes, and known defects automatically — remove the manual data-gathering step entirely.
2
Run the relevant analysis agents in parallel and synthesise their findings into one structured report: root cause, workaround, prioritised actions, and a confidence score.
3
Let engineers interrogate and refine that report through a conversational AI interface, with every edit tracked as a new version so the original AI output is never lost.
4
Support multiple LLM providers (Anthropic, OpenAI, Google, AWS Bedrock) interchangeably via configuration — zero agent code changes to switch.
5
Generate audience-specific exports (technical PDF, executive PDF/PPT, customer PPT) from the same underlying report data, removing duplicate manual write-ups.

Flows
Incident Intake
Engineers submit manually via the UI form, or incidents are auto-ingested from Jira via webhook. Both paths converge on the same pipeline.

🖥️ Path A — Manual (UI Form)

1
Engineer fills title, description, severity, metadata
2
POST /api/analysis returns job_id immediately
3
UI polls every 3 s while analysis runs in background
4
Report appears in UI on completion

⚡ Path B — Automatic (Webhook)

1
Jira creates or transitions a ticket matching trigger rules
2
Jira fires webhook → HMAC signature validated
3
Adapter maps payload to Incident model → Celery task enqueued
4
Slack notification sent with deep link to /analysis/{id}
Trigger rules are config-driven in config.yaml — only specific issue types, priorities, and project keys fire analysis, keeping noise low.
For this demo: incidents are created through the UI form — that's the only exercised end-to-end path. Jira/generic webhooks and Slack notification are real too, but Salesforce intake, Email/Webex notification, the workaround approval gate, and the evaluator agent are intentionally left as stubs that demonstrate the pluggable adapter architecture — see the Challenges & Risks section below.

Capabilities
Key Features
Beyond automated analysis — a complete engineer review workflow with AI collaboration built in.
🤖

Multi-Agent Pipeline

Orchestrator LLM decides which agents to activate. Log, code, and defect agents run in parallel via LangGraph's Send API. Synthesiser produces root cause, workaround, and prioritised actions with a confidence score.

💬

Chat for Clarifications

Engineers ask follow-up questions with full incident context — raw sub-agent findings, current report, and history all in the system prompt. Responses stream via SSE. AI can propose direct field edits (accept / dismiss).

📝

Analysis Versioning

Every edit — manual or chat-suggested — creates a new version. The original AI output (v0) is never overwritten. Version history with diffs is available in the UI. Share and export always use the latest approved version.

📈

Token Usage Tracking

Each analysis records per-agent token counts and estimated cost. Aggregate trends identify which agents are most expensive, enabling targeted prompt optimisation over time.

🔌

Pluggable Adapters

Every external system sits behind an abstract base class. Log Bundle, Jira, and Slack are real, wired examples; Splunk, Datadog, Salesforce, and Email/Webex are extensibility hooks — add a real one by subclassing one file and adding one config.yaml entry, no pipeline changes.

🔬

Evaluation Agent Extensibility hook

run_evaluation() defines the interface for an on-demand, post-closure agent that would score the prior AI analysis against the actual resolution and emit prompt-improvement hints — currently a documented stub (raises NotImplementedError), showing where that capability would attach.

Retrospective
Challenges & Risks
Problems hit during development and how each was mitigated, plus known risks carried forward.
🧩

LLM Output Reliability

Agents and the synthesizer depend on the LLM returning well-formed JSON. A malformed response could crash the whole analysis. Mitigation: regex extraction plus a try/except fallback that degrades to a low-confidence prose result instead of failing the task (src/agents/synthesizer.py).

🔀

Parallel State Merging

Multiple agents write findings concurrently during LangGraph's fan-out step; naive state merging can drop or overwrite results. Mitigation: Annotated[list[SubReport], operator.add] on the shared state so LangGraph merges sub-reports safely.

🔌

Multi-Service Local Environment

Running Postgres/Redis/API alongside other local projects on one machine caused port collisions and a misconfigured broker URL that silently crashed the Celery worker for weeks without anyone noticing. Mitigation: documented port remap in docker-compose.yml and .env.example, standardised on Docker-internal service hostnames for the containers.

🔄

Provider Portability

Supporting four LLM providers without agent code changes risked drift and duplicated logic. Mitigation: a single factory (src/core/llm.py) behind get_llm() — only .env changes to switch providers.

🗄️

Schema Evolution Without Downtime

The app originally relied on create_tables() at startup with no tracked migration history, making future schema changes risky. Mitigation: added a tracked Alembic baseline migration this cycle.

⚠️

Incomplete Integrations Risk Overstatement

Several integrations (Salesforce, Splunk/Datadog, Webex/Email, the workaround approval gate) are still stubs. Mitigation: tracked explicitly as Planned in PLAN.md and the Deliverables table above, rather than presented as complete.
